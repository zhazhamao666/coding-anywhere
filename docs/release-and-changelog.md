# 版本发布与变更记录规范

这份规范用于统一 `Coding Anywhere` 的版本发布方式，以及每次发布时的变更记录写法。

目标是解决 4 个问题：

- 什么时候应该发一个新版本
- 版本号应该怎么升
- CHANGELOG 应该怎么写
- 发布前后管理员该做哪些动作

当前项目的运行边界和验证入口以 [项目总览](./project-full-overview.md) 为准；部署、升级和回滚操作以 [管理员部署手册](./admin-deployment.md) 为准。

## 1. 适用范围

这份规范适用于：

- 本地测试版发布
- 内部试用版发布
- 管理员手动升级发布
- 文档版、修复版、功能版的变更记录整理

当前项目还没有正式的自动化发版流水线，所以这份规范默认按“手工发布”设计。

## 2. 总原则

### 原则 1：发布不是提交代码

不是每个 commit 都值得发版。

只有当一组变更已经：

- 可构建
- 可测试
- 可说明
- 可回滚

才建议视为一个“可发布版本”。

### 原则 2：CHANGELOG 写给人看

CHANGELOG 不是 git log 的复制品。

它要让管理员和使用者快速知道：

- 这次改了什么
- 有没有行为变化
- 需不需要特别关注升级风险

### 原则 3：发布前必须有验证证据

至少应看到：

- `npm run test` 通过
- `npm run build` 通过
- `npm run doctor` 通过
- 最小飞书联调通过

## 3. 推荐版本号规则

当前项目建议使用语义化版本：

```text
MAJOR.MINOR.PATCH
```

例如：

```text
1.0.0
1.1.0
1.1.1
2.0.0
```

### `PATCH`

适用于：

- bug 修复
- 重复回复修复
- 日志乱码修复
- 文档勘误
- 不改变使用方式的小修

例如：

- `1.0.0 -> 1.0.1`
- `1.0.1 -> 1.0.2`

### `MINOR`

适用于：

- 新增桥接命令
- 新增可见能力
- 新增管理员文档、测试剧本、排障能力
- 保持向后兼容的增强

例如：

- `1.0.2 -> 1.1.0`

### `MAJOR`

适用于：

- 修改用户使用方式
- 修改配置结构且不兼容旧配置
- 修改桥接协议行为
- 修改部署方式且旧方案不再适用

例如：

- `1.1.0 -> 2.0.0`

## 4. 什么情况下应该发版

建议满足以下任一条件就考虑发版：

- 修掉了一个真实影响使用的问题
- 新增了一个用户能直接感知到的功能
- 文档体系发生了显著完善，足以支撑交付
- 部署方式、配置方式、运维方式发生变化

不建议为下面这些情况单独发版：

- 只做了微小格式调整
- 中间态实验提交
- 未完成的半成品能力

## 5. 推荐发布节奏

当前阶段最适合的是“按里程碑发布”，不是“每改一点就发一个版本号”。

推荐节奏：

- 每完成一轮可演示、可回归的改动，发一个版本
- 一轮相关 bug 修复可以合并成一个 patch 版本
- 一组功能和文档补齐后发一个 minor 版本

## 6. CHANGELOG 建议存放位置

建议在项目根目录新增：

```text
CHANGELOG.md
```

如果你后续决定正式启用版本发布，推荐把每次版本摘要都写进这里。

在当前阶段，如果还不想维护根目录 `CHANGELOG.md`，也至少应保证：

- git commit 清晰
- README / docs 有同步更新
- 版本记录可追溯

## 7. CHANGELOG 的推荐结构

每个版本建议至少包含：

- 版本号
- 日期
- 变更摘要
- 破坏性变更
- 升级动作
- 验证结果

推荐模板：

```md
## 1.0.1 - 2026-03-22

### Added
- 新增 xxx

### Changed
- 调整 xxx

### Fixed
- 修复 xxx

### Docs
- 更新 xxx 文档

### Breaking Changes
- 无

### Upgrade Notes
- 升级后重新执行 `npm run build`
- 重启桥接服务

### Verification
- `npm run test`
- `npm run build`
- `npm run doctor`
- 飞书最小联调通过
```

## 8. CHANGELOG 条目怎么写

### 写法要求

每一条都尽量满足：

- 用过去式或结果式表达
- 说明“用户受影响的是什么”
- 不只写技术细节，还要写行为变化

### 好的写法

- 修复飞书普通消息被重复回复的问题
- 修复 Windows 控制台中文日志乱码问题
- 新增飞书联调测试剧本与故障排查手册
- 调整纯文本短回复逻辑，避免进度消息与最终消息重复显示

### 不好的写法

- fix bug
- update code
- improve logging
- misc changes

## 9. 推荐分类

建议统一使用这些分类：

- `Added`
- `Changed`
- `Fixed`
- `Docs`
- `Ops`
- `Breaking Changes`

### `Added`

新能力、新命令、新文档入口。

### `Changed`

行为调整、启动方式变化、默认流程变化。

### `Fixed`

实际 bug 修复。

### `Docs`

新增或完善说明文档。

### `Ops`

部署、运维、启动脚本、监控、日志相关变化。

### `Breaking Changes`

会影响旧使用方式的改动。

## 10. 发布前检查清单

每次准备发布前，建议按下面顺序检查：

1. 工作树干净
2. 版本说明已整理
3. 文档已更新
4. 执行：

```bash
npm run doctor
npm run build
npm run test
```

5. 启动服务：

```bash
npm run start
```

6. 飞书最小回归：

- `/ca status`
- `test`
- `/ca session`

7. 若本次有飞书 UI 或真实链路相关变更，再补对应 `coding-anywhere-autotest` 夹具 smoke。

## 11. 发布动作建议顺序

建议按这个顺序走一次发布：

1. 合并本轮代码
2. 更新文档
3. 整理 CHANGELOG
4. 跑测试与构建
5. 做最小飞书联调
6. 打版本标签或记录版本号
7. 通知使用者升级

## 12. 发布后检查清单

发布完成后，管理员至少确认：

- 线上或当前运行实例已是最新版本
- `readyz` 正常
- 飞书 `test` 只回复 1 次
- `/ca status` 正常
- 版本说明已经归档

## 13. 回滚时的变更记录要求

如果一次发布失败并回滚，建议在变更记录里明确写出：

- 回滚到哪个版本
- 为什么回滚
- 哪个问题触发了回滚
- 后续计划是什么

推荐写法：

```md
## 1.0.2 - 回滚说明

- 已回滚到 1.0.1
- 原因：飞书普通消息出现重复回复
- 影响：桥接服务对外表现不稳定
- 后续：修复后重新发布 1.0.2 或直接发布 1.0.3
```

## 14. 推荐的 git 提交与版本记录关系

推荐做法是：

- commit 负责记录“单次变更”
- changelog 负责记录“一个版本”

也就是说：

- 多个 commit 可以对应一个版本
- 一个版本不等于一个 commit

## 15. 当前项目的推荐发版粒度

结合这个项目的现阶段，我建议：

- 日常修 bug：按 patch 发版
- 新增桥接功能：按 minor 发版
- 改配置结构或部署方式：按 major 发版

### 适合归入 patch 的例子

- 重复回复修复
- 控制台乱码修复
- Codex CLI JSONL 解析兼容修复

### 适合归入 minor 的例子

- 新增 `/ca` 命令
- 新增日志查询能力
- 新增飞书卡片、ops 或 live smoke 能力

### 适合归入 major 的例子

- 改掉当前配置文件结构
- 引入新的消息通道协议
- 从单实例模式变成另一套运行模型

## 16. 一份适合当前项目的首版 CHANGELOG 示例

## 17. 最推荐的落地方式

如果你后续准备把这套规范真正执行起来，我建议下一步做两件事：

1. 在项目根目录新增 `CHANGELOG.md`
2. 每次对外可用版本都按这份规范补一条版本记录

这样你后面做升级、回滚、交接时会轻松很多。
